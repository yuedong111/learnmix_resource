# 工程目录设计任务：知识库 RAG 项目目录结构设计

## 任务目标
请基于项目类型、业务场景和当前技术栈，设计一套清晰、可扩展、便于协作和便于 AI/开发者理解的工程目录结构，并说明各目录职责与边界。

## 背景上下文
当前项目处于以下阶段：
- 项目类型：企业知识库 / RAG / 文档处理系统
- 业务域：文档上传、解析、切分、向量入库、问答、FAQ 生成、审核
- 当前技术栈：Python、FastAPI、Celery、Redis、PostgreSQL、Elasticsearch、Milvus
- 当前规模：中型后端项目，多个服务模块
- 团队规模：3-8 人
- 预期演进方向：支持更多文档类型、更多异步任务、更多知识加工能力

## 输入材料
请参考以下信息：
- 核心业务模块：
  - 文档接入
  - 文档解析
  - chunk 管理
  - embedding 入库
  - 检索与问答
  - FAQ 生成
  - 审核与发布
- 已有代码现状：
  - service、script、task、api 代码混放
  - 一些业务逻辑写在脚本里
  - 测试结构不清晰
- 典型开发任务：
  - 新增一个异步任务
  - 新增一个后台接口
  - 新增一个 repository
  - 新增一个数据处理脚本
- 部署方式：
  - API 服务 + Celery worker 分开部署
- 是否包含异步任务/脚本/接口/前端：
  - 包含异步任务、管理接口、离线脚本
- 当前主要问题：
  - 新代码不知道该放哪里
  - 业务逻辑在多个目录重复
  - agent 改代码时容易找错位置

## 约束条件
请遵守以下要求：
- 目录结构要匹配 Python 后端项目
- 不要过度分层
- 明确 api / service / repository / task / script 的职责边界
- 说明哪些适合放 domain 层
- 说明测试目录如何组织
- 优先考虑可维护性、可测试性、可协作性

## 输出格式
请按以下结构输出：
1. 目录设计目标
2. 推荐目录结构树
3. 每个目录的职责说明
4. 典型代码放置示例
5. 依赖方向建议
6. 常见反模式
7. 演进建议

## 验收标准
输出结果应满足：
- 开发者可以快速判断代码该放在哪
- 可以避免脚本层和业务层混乱
- 能支持后续持续扩展
- 对 agent 理解仓库结构友好
