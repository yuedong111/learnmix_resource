# Chapter 10: 设计模式


在上一章，我们学习了软件架构设计，了解了如何规划系统的整体结构。但光有架构蓝图还不够——当需要实现具体功能时，我们常常面临代码混乱、难以复用的问题。比如，如何确保一个类只有一个实例？如何让不同系统协同工作？这就是本章要介绍的**设计模式**，它就像软件开发者的"工具箱"，提供经过验证的解决方案，让代码更清晰、更易维护。

## 10.1 为什么需要设计模式？

想象一下，你要开发一个电商网站，需要实现用户登录、商品展示、订单处理等功能。如果没有设计模式，可能会出现：
- 用户登录类创建多个实例，导致状态混乱
- 商品展示和订单处理系统耦合紧密，修改一个功能影响另一个
- 新功能添加时，需要大量修改现有代码

设计模式就是解决这些问题的"标准工具"。就像建筑工人用榫卯结构连接木材，设计模式提供了一套"连接代码"的规范，让系统更稳定、更易扩展。

## 10.2 设计模式是什么？

根据源材料，设计模式是"**对被用来在特定场景下解决一般设计问题的类和互相通信的对象的描述**"。简单来说：
- **解决特定问题**：每种模式针对一类常见问题（如单例模式解决"唯一实例"问题）
- **通用解决方案**：不是具体代码，而是可复用的思想（如工厂模式可以创建不同类型的对象）
- **提高复用性**：像工具箱里的扳手，遇到类似问题就能用

> 源材料提到："设计模式是软件架构设计师的必修课，设计模式中蕴含的思想是架构设计师必须掌握的。"

## 10.3 设计模式的分类

设计模式主要分为三类，就像工具箱里的不同工具：

### 1. 创建型模式（解决"如何创建对象"）
- **单例模式**：确保一个类只有一个实例（如系统配置类）
- **工厂模式**：封装对象创建过程（如根据用户类型创建不同账户）
- **建造者模式**：逐步构造复杂对象（如组装电脑）

### 2. 结构型模式（解决"如何组织类"）
- **适配器模式**：让不兼容的接口协同工作（如旧系统和新系统对接）
- **装饰器模式**：动态扩展功能（如给咖啡加糖、加奶）
- **外观模式**：简化复杂系统接口（如电脑的电源按钮）

### 3. 行为型模式（解决"对象如何交互"）
- **观察者模式**：对象状态变化时通知其他对象（如股票价格变动通知）
- **策略模式**：算法可互换（如不同支付方式）
- **中介者模式**：减少对象间耦合（如聊天室中的消息转发）

## 10.4 常见设计模式详解

### 10.4.1 单例模式（Singleton）

**解决的问题**：确保一个类只有一个实例，避免状态混乱。

**例子**：系统日志类，如果创建多个实例，日志会重复记录。

```java
// 单例模式实现
public class Logger {
    private static Logger instance; // 静态实例
    
    private Logger() {} // 私有构造方法
    
    public static Logger getInstance() {
        if (instance == null) {
            instance = new Logger(); // 延迟初始化
        }
        return instance;
    }
    
    public void log(String message) {
        System.out.println("日志: " + message);
    }
}

// 使用示例
Logger logger1 = Logger.getInstance();
Logger logger2 = Logger.getInstance();
System.out.println(logger1 == logger2); // 输出: true
```

**效果**：保证全局唯一实例，但过度使用会降低灵活性。

### 10.4.2 工厂模式（Factory）

**解决的问题**：封装对象创建过程，避免硬编码类名。

**例子**：根据用户类型创建不同账户（普通用户/管理员）。

```java
// 产品接口
interface Account {
    void login();
}

// 具体产品
class UserAccount implements Account {
    public void login() { System.out.println("用户登录"); }
}

class AdminAccount implements Account {
    public void login() { System.out.println("管理员登录"); }
}

// 工厂类
class AccountFactory {
    public static Account createAccount(String type) {
        if ("admin".equals(type)) {
            return new AdminAccount();
        } else {
            return new UserAccount();
        }
    }
}

// 使用示例
Account user = AccountFactory.createAccount("user");
Account admin = AccountFactory.createAccount("admin");
```

**效果**：灵活创建对象，但工厂类可能变得复杂。

### 10.4.3 观察者模式（Observer）

**解决的问题**：对象状态变化时通知其他对象，实现松耦合。

**例子**：股票价格变动时通知所有关注者。

```java
// 观察者接口
interface Observer {
    void update(float price);
}

// 被观察者
class Stock {
    private List<Observer> observers = new ArrayList<>();
    private float price;
    
    public void addObserver(Observer o) { observers.add(o); }
    public void setPrice(float price) {
        this.price = price;
        notifyObservers();
    }
    
    private void notifyObservers() {
        for (Observer o : observers) {
            o.update(price);
        }
    }
}

// 具体观察者
class Investor implements Observer {
    public void update(float price) {
        System.out.println("股价变动: " + price);
    }
}

// 使用示例
Stock stock = new Stock();
stock.addObserver(new Investor());
stock.setPrice(100.5f); // 触发通知
```

**效果**：实现广播通知，但观察者过多可能影响性能。

## 10.5 设计模式的使用原则

1. **按需使用**：不是所有问题都需要设计模式，简单问题用简单方法
2. **理解思想**：模式是工具，不是教条（如MVC既是模式也是架构风格）
3. **组合使用**：复杂系统常结合多种模式（如Spring框架同时使用工厂和代理模式）

> 源材料提醒："滥用设计模式会适得其反。在软件设计中使用设计模式，可以优化设计，提高架构质量。但是...不相宜的场合乱用设计模式有害无益"

## 10.6 设计模式与架构的关系

设计模式是架构的"微观工具"，架构是系统的"宏观蓝图"。就像：
- 架构设计决定用分层系统（如表示层/业务层/数据层）
- 设计模式决定每层内部如何组织（如业务层用策略模式处理不同算法）
